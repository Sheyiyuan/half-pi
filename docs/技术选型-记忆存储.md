# 技术选型：记忆存储

> 评估日期：2026-05-21  
> 关联：[记忆系统设计](./memory-design.md)

half-pi 的记忆系统需要在「检索能力」和「云端同步」之间做权衡。

## 决策矩阵

核心矛盾就一个：

```
高效检索  ←──────────────────────→  云端友好
  SQLite+vec                            Markdown
  ChromaDB                              JSONL
  LevelDB

            双轨制 ← 折中方案
            Markdown 存原文 + SQLite 存索引
```

### 为什么不是纯 SQLite

- 二进制 blob，git diff 不可用
- 多设备同步需要手动合并 `.db` 文件，冲突概率极高
- Agent 直接写 SQL 容易出错（schema 变更、注入风险）

### 为什么不是 JSONL

- 没有结构化字段，检索靠 grep 遍历
- 大量条目时线性扫描不可接受
- 适合 append-only 场景（如 session log），不适合随机存取

### 为什么不是 ChromaDB

- 需要额外 Python 进程或嵌入 runtime
- 功能过剩（专业 RAG 框架）
- 部署复杂度高

### 为什么是双轨制

- Markdown 的 git 友好性完美匹配「云端同步」需求
- SQLite+vec 的检索能力完美匹配「语义搜索」需求
- index 丢了可以重建（从 entries/ 重新 embedding）
- 职责分离：entries 管内容，index 管检索

---

## 选定方案：双轨制

```
~/.half-pi/memory/
├── entries/              ← Markdown 原文，git 同步
│   ├── local/
│   │   └── env-*.md      # 设备绑定
│   └── cloud/
│       └── pref-*.md     # 云端共享
│
└── index.sqlite          ← 本地索引，不进 git
    ├── entries           # 元数据表
    ├── embeddings        # 向量表（Phase 5）
    └── stats             # 调用统计表
```

| | entries/ | index.sqlite |
|---|---|---|
| 存储内容 | 记忆原文 | 元数据 + 向量 + 统计 |
| 进 git？ | ✅ | ❌ (.gitignore) |
| 谁写入？ | Agent 工具 | 自动维护 |
| 丢了怎么办？ | git restore | `npm run memory:reindex` |

---

## 阶段路线

| 阶段 | 检索方式 | 存储 |
|------|---------|------|
| **Phase 1-2**（当前） | 全量注入 + ripgrep + 权重排序 | 纯 Markdown |
| **Phase 3-4** | 同上 + 自整理 | 纯 Markdown |
| **Phase 5** | embedding + cosine similarity | Markdown + sqlite-vec 索引 |

Phase 1-4 不需要向量检索——量级在 200 条以内时全量注入 + YAML 过滤足够。Phase 5 是远期升级。

---

## 为什么现在选 Markdown

1. **和现有系统一致。** SOUL.md 和 skills 已经是 Markdown+YAML，记忆走同样的路零认知负担
2. **git 友好。** 云端同步不需要任何额外工具
3. **Agent 自管理。** `skill_create` 的工具代码改几行就是 `memory_create`
4. **渐进升级。** `MemoryStore` 接口预留 `search(query, limit)`，将来换 sqlite-vec 后端不影响上层
