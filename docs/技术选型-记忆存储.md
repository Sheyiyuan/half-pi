## 决策矩阵

核心矛盾：

```
高效检索  ←──────────────────────→  云端友好
  SQLite+vec                            Markdown
  ChromaDB                              JSONL
  LevelDB

          双轨制 ← 折中方案
          Markdown 存原文 + SQLite 存索引（远期）
```

### 淘汰方案

| 方案 | 淘汰原因 |
|------|---------|
| **纯 SQLite** | 二进制 blob，git diff 不可用；多设备同步冲突概率极高；Agent 写 SQL 容易出错 |
| **JSONL** | 无结构化字段，大量条目时线性扫描不可接受；适合 session log，不适合随机存取 |
| **ChromaDB** | 功能过剩（专业 RAG 框架）；需额外 Python 进程；部署复杂 |
| **LevelDB** | 同样是二进制 KV，git 不友好；没有额外的语义检索优势 |

### 选定方案：双轨制

| | entries/ | index.sqlite（远期） |
|---|---|---|
| 存储内容 | 记忆原文 | 元数据 + 向量 + 统计 |
| 进 git？ | ✅ | ❌ (.gitignore) |
| 谁写入？ | Agent 工具 | 自动维护 |
| 丢了怎么办？ | git restore | `npm run memory:reindex` |

选 Markdown 的三个核心原因：

1. **和现有系统一致。** SOUL.md 和 skills 已经是 Markdown+YAML，记忆走同样的路零认知负担
2. **git 友好。** 云端同步不需要任何额外工具
3. **Agent 自管理。** `skill_create` 的工具代码改几行就是 `memory_create`

## 远期：sqlite-vec 向量索引

Phase 5 才会引入（量级 < 200 条时全量注入足够）。升级策略：

```
Phase 1-4                Phase 5
纯 Markdown (git)  ──→   Markdown (git) + sqlite-vec (本地)

新增文件：
  ~/.half-pi/memory/index.sqlite/
  ├── entries 表         # 元数据镜像
  ├── embeddings 表      # 向量（通过 embedding 模型生成）
  └── stats 表           # 调用统计

重建命令：half-pi memory reindex（从 entries/ 重新 embedding 所有记忆）
```

接口层 `MemoryStore` 预留 `search(query, limit)` 方法，将来换后端不影响上层。
